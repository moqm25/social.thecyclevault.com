# SCALING_PLAN.md — The CycleVault Social

- **Status:** Phase 0 spec (Accepted)
- **Date:** 2026-06-26
- **Owner:** [`adr/0001-foundational-architecture.md`](./adr/0001-foundational-architecture.md)

How the platform grows from launch to 100k+ users without a re-platform, and where
the known Firestore limits bite. Pairs with [`COST_MODEL.md`](./COST_MODEL.md).

---

## 1. Scaling stages

### Stage 1 — 0 to 10k users (launch)

- **Entirely Firebase**, free or near-free.
- Firestore search = simple field queries + composite indexes (`DATA_MODEL.md` §14).
- Single default Firestore database; counters denormalized; reads cached client-side.
- **No premature optimization.** Distributed counters and external search are _not_
  needed yet — design leaves room for them but they stay off.

### Stage 2 — 10k to 100k users

- Add an **external search** provider (Algolia or Meilisearch) — Firestore can't do
  full-text search (§4).
- Add **distributed/sharded counters** on hot documents (§3).
- Add **caching** at the edge for hot read paths (e.g. cached "hot" feed snapshots
  regenerated by a scheduled function).
- Introduce **background jobs** (scheduled functions) for ranking recomputation,
  digest notifications, and cleanup.
- Revisit indexes against real query metrics; prune unused ones.

### Stage 3 — 100k+ users

- Evaluate a **dedicated API layer** (Cloud Run) for the heaviest read paths and
  response shaping, with Firestore still as the store.
- Consider **read replicas of denormalized feeds** (precomputed feed documents),
  CDN caching, and event-driven aggregation pipelines.
- Re-examine data partitioning and whether any collection should move to a
  purpose-built store. The `lib/firebase` isolation layer (ADR-0001 §4.3) keeps this
  tractable.

## 2. Firestore limits to design around

| Limit                                    | Value                     | Implication                                                |
| ---------------------------------------- | ------------------------- | ---------------------------------------------------------- |
| Sustained writes **per single document** | ~**1/sec**                | Hot counters (popular post score) need sharding            |
| Max document size                        | **1 MiB**                 | Don't embed unbounded arrays (e.g. all comments in a post) |
| Writes in a transaction/batch            | **500**                   | Bulk ops must chunk                                        |
| Index entries per document               | bounded                   | Avoid over-indexing large array fields                     |
| Query returns                            | only what indexes support | No full-text; no cross-collection joins                    |

The model in `DATA_MODEL.md` already avoids unbounded embedding (comments are their
own collection; counters are scalars).

---

## 3. Hot-document mitigation (the #1 scaling risk)

A viral post receiving many votes/comments per second exceeds the **~1 write/sec per
document** ceiling on its counter fields.

**Mitigations, applied progressively:**

1. **`FieldValue.increment`** (Stage 1) — avoids read-modify-write contention for
   ordinary traffic; sufficient for most posts.
2. **Sharded counters** (Stage 2) — split a hot counter across _N_ shard docs
   (`posts/{id}/shards/{0..N-1}`); writes hit a random shard, reads sum shards. Apply
   only to documents that demonstrably run hot.
3. **Buffered aggregation** (Stage 2–3) — write raw vote events; a scheduled/triggered
   function rolls them into the displayed score every few seconds.

`hotRank` recomputation likewise moves from inline (Stage 1) to batched
(Stage 2+) for hot items.

## 4. Search

- **Stage 1:** Firestore queries only — browse by community, sort Hot/New/Top, filter
  by tag, exact lookups (username, community). **No full-text search.**
- **Stage 2:** integrate **Algolia** or **Meilisearch**. A Firestore trigger syncs
  `posts` (and optionally `users`, `communities`) into the search index on
  write/remove. Search targets: posts, users, communities. Keep only public,
  non-sensitive fields in the index.
- Self-hosting Meilisearch trades a server to operate for lower cost; Algolia is
  zero-ops with a free tier. Decide at Stage 2 based on volume.

---

## 5. Read-scaling levers (recap, ordered by impact)

1. Client cache + dedupe (TanStack Query).
2. Pagination/cursors everywhere — bounded reads.
3. Denormalized counters (no count-on-read).
4. Precomputed feed snapshots for the default "Hot" view (Stage 2).
5. CDN/edge caching of public, cacheable responses (Stage 3).

---

## 6. Notifications at scale

- **Stage 1:** top-level `notifications` collection, indexed by `recipientId`.
- **Stage 2+:** if per-user volume or read-isolation becomes a concern, migrate to
  **`users/{uid}/notifications`** subcollections (cleaner security, natural sharding
  by user) and/or add web push (FCM). Fan-out for very high-degree authors uses a
  queue + batched writes to respect the 500-write batch limit.

---

## 7. Functions at scale

- Watch **cold starts** on write paths; set **min instances** only on functions that
  prove latency-sensitive (cost trade-off).
- Keep functions **single-purpose and idempotent**.
- Move periodic work (ranking, digests, cleanup, search sync) to **scheduled /
  triggered** functions rather than doing it inline on the request path.

---

## 8. Triggers for scaling decisions (watch these metrics)

| Signal                                            | Action                                       |
| ------------------------------------------------- | -------------------------------------------- |
| Firestore reads approaching free tier daily       | Tighten caching/pagination first             |
| A post regularly exceeds ~1 write/sec on counters | Turn on sharded counters for hot docs        |
| Users ask for real search / search feels weak     | Stand up Algolia/Meilisearch (Stage 2)       |
| p95 read latency rising on feeds                  | Precompute feed snapshots; add caching       |
| Function cold-start latency hurts UX              | Min instances on the hot function(s)         |
| Monthly bill trending up super-linearly           | Profile top read paths; consider Stage 3 API |

---

## 9. Non-goals (avoid premature complexity)

No Kubernetes, no microservices, no multi-region replication, no dedicated API at
launch. These are Stage 3 _evaluations_, not Stage 1 commitments. The cheapest
scaling is the work we **don't** do until metrics demand it.
